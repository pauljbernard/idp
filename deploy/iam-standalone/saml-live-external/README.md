# SimpleSAMLphp Live External Target

This directory provides a concrete local bootstrap path for the first live external SAML SP lane.

It is intentionally narrow:

- SP family: `SimpleSAMLphp`
- bounded profile: `saml-sp-bounded-redirect-post-v1`
- request binding: `REDIRECT`
- response binding: `POST`

## Files

- `simplesamlphp.env.example`
- `docker-compose.simplesamlphp.yml`
- `simplesamlphp-authsources.php`
- `simplesamlphp-saml20-idp-remote.php`

## Expected flow

1. Start the IDP API on `http://127.0.0.1:4000`.
2. Copy `simplesamlphp.env.example` to `.env` in this directory and adjust ports if needed.
3. Start SimpleSAMLphp:

```bash
npm run start:idp:saml-live-external-simplesamlphp
```

4. Verify the container and metadata endpoint:

```bash
npm run verify:idp:saml-live-external-simplesamlphp
```

5. Verify the live external environment contract:

```bash
export IDP_SAML_SP_TARGET_BASE_URL=http://127.0.0.1:4000
export IDP_SAML_SP_CLIENT_ID=saml-test-service-provider-live-local
export IDP_SAML_SP_METADATA_URL=https://127.0.0.1:19443/simplesaml/module.php/saml/sp/metadata.php/default-sp
npm run verify:idp:saml-live-external-env
```

6. Execute the live external lane:

```bash
npm run test:integration:bounded-saml-sp-live-external-target
```

7. Stop the SP when finished:

```bash
npm run stop:idp:saml-live-external-simplesamlphp
```

## Boundary

This is a bootstrap template, not yet a passing external artifact. The lane only becomes real evidence when:

- the SimpleSAMLphp container is actually running,
- metadata is fetched live,
- and `latest-bounded-saml-sp-live-external-target.json` is generated by a real run.

## Client Contract

The live external bootstrap uses the dedicated loopback SAML client:

- `client_id`: `saml-test-service-provider-live-local`

That client is intentionally separate from the synthetic fixture client so the runtime can keep exact ACS URL matching enabled while admitting real loopback ACS endpoints for bounded external validation.
